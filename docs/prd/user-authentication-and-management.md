# 用户管理系统：完整认证与用户生命周期

## Problem Statement

作为一个教育平台产品，当前系统的用户管理仅停留在 email 查找用户 + localStorage 模拟 session 的 demo 级别。页面刷新后认证状态丢失，没有密码基础设施，没有 email 验证，没有密码重置，没有管理员用户管理面板，没有用户个人中心。这些缺失使产品无法用于真实用户场景。

## Solution

集成 Better Auth 作为认证底座，构建完整的用户生命周期管理：注册 → 邮箱验证 → 角色选择 → 日常登录 → 密码重置 → 个人资料管理 → 管理员用户管理。所有功能基于 HTTP-only Session Cookie 实现安全认证，替代现有的 localStorage 方案。

## User Stories

1. 作为访客用户，我希望无需注册即可浏览基本内容，以便快速了解产品价值。
2. 作为新用户，我希望通过 email 和密码注册账号，以便拥有个人账户。
3. 作为新用户，我希望在注册后收到验证邮件并点击链接完成验证，以便激活完整功能。
4. 作为新用户，我希望在验证邮箱后被引导选择角色（学生/老师/辅导员/家长/其他），以便获得个性化的产品体验。
5. 作为学生，我希望填写学校名称和毕业年份，以便获得针对我背景的推荐。
6. 作为老师，我希望填写任教学科，以便看到与我相关的专业信息。
7. 作为辅导员，我希望填写专业领域，以便管理我负责的学生群体。
8. 作为已注册用户，我希望通过 email 和密码登录，以便访问我的个人数据和收藏。
9. 作为已登录用户，我希望关闭浏览器后重新打开仍能保持登录状态，以便不用重复登录。
10. 作为已登录用户，我希望在导航栏看到我的名字/头像，点击后弹出个人菜单，以便快速访问账号设置。
11. 作为忘记密码的用户，我希望通过 email 收到重置链接，以便重新设置密码。
12. 作为忘记密码的用户，我希望重置链接在 1 小时后失效，以便保证安全。
13. 作为密码重置后的用户，我希望所有已登录设备被踢出，以便确保盗用者无法继续访问。
14. 作为已登录用户，我希望修改密码（需输入当前密码），以便保障账号安全。
15. 作为已登录用户，我希望修改个人资料（名称、学校、年级等），以便保持信息准确。
16. 作为已登录用户，我希望修改我的用户类型（如从学生改为老师），以便反映我的身份变化。
17. 作为已登录用户，我希望查看我的订阅状态和到期时间，以便了解我的权益。
18. 作为已登录用户，我希望查看和管理我的收藏（大学、专业），以便快速找到关注的内容。
19. 作为已登录用户，我希望删除我的账号（带确认弹窗），以便行使数据删除权。
20. 作为管理员，我希望查看所有用户列表（搜索、筛选、分页），以便管理用户群体。
21. 作为管理员，我希望查看单个用户的完整信息，以便了解用户状态。
22. 作为管理员，我希望修改用户的权限角色（FREE/PRO/ADMIN），以便控制功能访问。
23. 作为管理员，我希望禁用/启用用户账号，以便处理违规行为。
24. 作为管理员，我希望人工重置用户密码，以便帮助用户恢复账号访问。
25. 作为管理员，我希望手动修改用户的订阅状态（激活/取消/trial），以便处理特殊情况。
26. 作为管理员，我希望调整用户的订阅到期时间，以便延长或缩短权益期限。
27. 作为管理员，我希望给用户发放试用（trial），以便促进转化。
28. 作为管理员，我希望查看用户的订阅历史，以便追踪订阅变更轨迹。
29. 作为产品经理，我希望区分 role（付费权限层）和 userType（产品体验层），以便独立控制功能访问和个性化体验。
30. 作为存量用户（迁移前已有的 email 记录），我希望首次登录时设置密码，以便平滑过渡到新认证体系。
31. 作为 demo 用户，我希望使用 demo@college.edu 快速绕过注册流程，以便演示产品功能。

## Implementation Decisions

### 认证库选择

采用 **Better Auth** 作为认证基础设施。TypeScript-first 的现代认证库，原生支持 Express、email/password、session cookie、password reset、email verification、rate limiting，不需要独立服务进程。作为 npm 包直接集成到现有 `server/server.ts` BFF 中。

### 认证方式

Email + Password + HTTP-only Session Cookie。不使用 JWT（localStorage 方案），不使用 OAuth（预留接口，暂不实现）。

### 密码策略

- Hash 算法：bcrypt
- 最小长度：8 位
- 密码重置：email 发送一次性 token 链接，有效期 1 小时
- 重置成功后：踢掉该用户所有 session

### 角色模型（双层设计）

- `role`：付费/功能权限层（GUEST | FREE | PRO | ADMIN），由管理员控制
- `userType`：产品体验层（STUDENT | TEACHER | COUNSELOR | PARENT | OTHER），用户可自行修改

### 注册流程（4 步）

1. 填写 email + password → 发送验证邮件
2. 点击邮件验证链接 → 激活账户
3. 登录后引导选择角色 → 决定 userType
4. 填写角色专属字段（学校/学科等）

未验证 email 的用户无法访问产品功能页面。例外：`demo@college.edu` 保持 bypass。

### 数据库迁移

在现有 User 模型上新增：`passwordHash`（NOT NULL）、`emailVerified`（default false）、`emailVerifiedAt`、`lastLoginAt`、`disabled`（default false）。Better Auth 自动创建 session、account、verificationToken 等辅助表。

### 前端 UI 模式

- **登录/注册/密码重置**：独立页面
- **用户个人中心**：侧边抽屉/弹窗
- **管理员面板**：独立页面/tab（嵌入现有 IPEDS Admin 旁）
- **顶部导航栏**：右侧新增用户头像/名称 + 下拉菜单

### Admin 面板范围

用户列表（搜索/筛选/分页）、用户详情、角色修改、禁用/启用、密码重置、订阅管理（状态/到期时间/trial）。不包含支付网关集成。

## Testing Decisions

只测试外部行为，不测试内部实现。需要测试：认证端点（注册/登录/登出/密码重置/邮箱验证）、Session 管理（持久性、重置后失效）、路由守卫（未登录重定向、非管理员拒绝）、Entitlement 映射、管理员操作。建议采用 Supertest + Vitest 对 BFF 端点进行集成测试，React Testing Library 对前端组件进行单元测试。项目目前无测试文件，这是第一个引入测试体系的功能。

## Out of Scope

- 第三方 OAuth 登录（Google/GitHub/Apple）— 预留接口
- 双因素认证（2FA）— 暂不需要
- 邀请码/推荐系统 — 已有独立 issue（#87）
- 支付网关集成（Stripe webhook）— 由 issue #86 覆盖
- 多租户/组织管理 — 预留
- 用户头像上传 — 暂不支持
